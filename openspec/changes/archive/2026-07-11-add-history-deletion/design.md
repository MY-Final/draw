# Design — add-history-deletion

## 背景约束

会话不是一张表,而是从 `generations.params.conversationId` 派生的视图(见 `conversations.js`)。素材(Blob)与生成元数据分离存储,以 id 互引。删除必须同时尊重这两条既有事实。

## 决策

### D1 — 连带删除:引用感知,而非只看收藏

用户选择"未收藏才连删",但仅看收藏不安全:产出图可能被别的会话当参考图引用("继续创作"=拿产出图当参考)。

判定一张产出图 `img` 可连带删除,当且仅当:

```
img.favorite === false
  且
不存在任何"删除后仍存活"的 generation,其 refImageIds 包含 img
```

任一条件不满足即保留。参考图 `refImageIds` 永不参与连带删除。

- **Why**:漏掉引用检查会让 B 会话的参考图缩略在删 A 会话后变裂图。
- **实现**:批量删除时先算出"删除后存活集合",再对候选产出图逐一检查是否仍被引用。收藏判定读 asset.favorite。

### D2 — 删除粒度与���认策略

| 操作 | 触发位置 | 防误 | 落库时机 |
|------|----------|------|----------|
| 删单条生成 | 回复卡操作区 | 可撤销 toast | 延迟提交(见 D3) |
| 删整段会话 | 左侧会话项菜单 | 确认弹窗 | 立即 |
| 清空全部 | 存储与备份抽屉 | 确认弹窗(标"不可逆") | 立即 |

- **Why**:单条误删最常见、损失最小 → 轻量可撤销;会话/清空损失大 → 强确认。

### D3 — 单条删除走延迟提交(非软删)

点删除时:立即从内存(store.generations)移除该条并渲染 toast;启动 setTimeout(约 5s);超时才真正 `deleteGeneration` + 连带删图。撤销 = 清除定时器 + 恢复内存条目。

- **Why**:纯前端小工具,撤销窗口仅数秒。软删要给数据模型加 `deletedAt` 并让所有查询过滤,成本不匹配收益。
- **代价**:撤销窗口内刷新页面,该条会"复活"(定时器随页面丢失,尚未落库删除)。可接受的极端边角。
- **一致性**:连带删图也放在超时回调里执行,撤销时无需回滚素材。

### D4 — 会话标题覆盖:localStorage 映射,不建表

会话仍是派生视图。手动重命名存 `localStorage['workbench.convTitles'] = {convId: title}`。`deriveConversations` 读该表,若存在覆盖标题则优先于派生的首句 prompt。删除会话时一并清除其覆盖项。

- **Why**:保持"会话=派生视图"的既有架构,避免为一个标题引入新存储表。

### D5 — 删除的连锁清理

删会话 = 删该会话所有 generation(按 D1 处理各自产出图)+ 清除其标题覆盖 + 若删的是当前会话则切到最近的其他会话(无则新建空会话)。清空全部 = 删所有 generation + 所有 asset + 清空标题覆盖表,保留 presets/activePresetId/Key。

### D6 — UI 走 ui-ux-pro-max

删除按钮、会话项菜单、确认弹窗、撤销 toast 的视觉走 `ui-ux-pro-max` 技能,沿用现有中性灰+蓝 token,SVG 图标(新增 trash 图标),不用 emoji。危险操作(清空全部)用告警色。

## 未决/非目标

- 不做批量多选删除(一次删多段会话)。
- 不做回收站/软删历史。
- 不做云端同步的删除传播(纯本地)。
